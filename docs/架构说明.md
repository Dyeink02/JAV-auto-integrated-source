# JAV自动化爬虫工具 架构说明

## 1. 整体架构

本项目采用 **Electron + TypeScript** 架构，分为三层：

```
┌──────────────────────────────────────────────────────────┐
│  表现层（Presentation Layer）                              │
│  desktop/renderer/  - HTML/CSS/JS 前端 UI                  │
│  职责：用户交互、状态展示、日志渲染、表单控制                │
└────────────────────┬─────────────────────────────────────┘
                     │ Electron IPC (invoke/handle)
┌────────────────────▼─────────────────────────────────────┐
│  业务层（Business Layer）                                  │
│  desktop/mainServices/ - Electron 主进程业务逻辑           │
│  职责：任务调度、设置管理、IPC 路由、日志桥接                │
└────────────────────┬─────────────────────────────────────┘
                     │ Child Process (stdio)
┌────────────────────▼─────────────────────────────────────┐
│  核心层（Core Layer）                                      │
│  src/core/ - TypeScript 爬虫核心（编译到 dist/）            │
│  职责：HTTP 请求、HTML 解析、队列编排、文件操作、Cloudflare │
└──────────────────────────────────────────────────────────┘
```

### 通信机制
| 层级间通信 | 方式 | 说明 |
|-----------|------|------|
| 表现层 ↔ 业务层 | Electron IPC (`invoke`/`handle`) | 沙盒模式下通过 preload.js 桥接 |
| 业务层 ↔ 核心层 | Child Process (stdio) | 爬虫核心作为子进程运行 |
| 业务层 → 前端 | IPC 事件 (`webContents.send`) | 流式推送进度、日志、状态 |

## 2. 爬虫核心架构（四阶段流水线）

```
                    ┌──────────────────┐
                    │  QueueManager    │
                    │  (阶段编排器)      │
                    └────────┬─────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
    ┌───────────┐    ┌──────────────┐    ┌──────────────┐
    │ Phase 1   │    │ Phase 2      │    │ Phase 3      │
    │ 索引页面  │───▶│ 详情页面     │───▶│ 文件写入     │
    │ (1x并发)  │    │ (0.75x并发)  │    │ (2x并发)     │
    └───────────┘    └──────────────┘    └──────────────┘
                                               │
                                               ▼
                                         ┌──────────────┐
                                         │ Phase 4      │
                                         │ 图像下载     │
                                         │ (0.5x并发)   │
                                         └──────────────┘
```

### 2.1 阶段职责

| 阶段 | 模块 | 并发 | 职责 |
|------|------|------|------|
| Phase 1 | `RequestHandler` | 1x | 抓取列表页面，解析出影片详情页链接 |
| Phase 2 | `Parser` | 0.75x | 解析详情页，提取元数据（gid、uc、img、title、分类、演员） |
| Phase 3 | `FileHandler` | 2x | 将元数据写入 JSON/CSV 文件（高优先级 I/O） |
| Phase 4 | `FileHandler` | 0.5x | 下载海报图片（带 referer，低优先级） |

### 2.2 核心模块

| 模块 | 路径 | 职责 |
|------|------|------|
| **QueueManager** | `src/core/queueManager.ts` | 四阶段流水线编排，任务分发，并发控制 |
| **RequestHandler** | `src/core/requestHandler.ts` | HTTP 客户端，带重试（axios-retry）、代理支持、UA 轮换 |
| **Parser** | `src/core/parser.ts` | Cheerio HTML 解析，多站点回退策略，元数据提取 |
| **PuppeteerPool** | `src/core/puppeteerPool.ts` | Chromium 浏览器实例池，Cloudflare 年龄验证绕过 |
| **Config** | `src/core/config.ts` | 多源配置管理（CLI → 缓存 → 系统代理 → 默认值） |
| **FileHandler** | `src/core/fileHandler.ts` | 文件读写、目录创建、批量写盘优化 |
| **Logger** | `src/core/logger.ts` | Winston 多级日志（debug/info/warn/error） |
| **FilmIdentity** | `src/core/filmIdentity.ts` | 番号提取与标准化（`normalizeFilmId`） |

## 4. 整理引擎架构

### 4.1 整理流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  扫描阶段   │───▶│  判定阶段   │───▶│  重命名阶段 │───▶│  报告阶段   │
│ scanPhase   │    │ judgePhase  │    │ renamePhase │    │ reportPhase │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
        │                  │                   │                  │
        ▼                  ▼                   ▼                  ▼
   遍历源目录         分类判定：            按番号重命名        生成 6 个 txt
   提取视频文件       - 待整理              或保留原名          报告文件
                      - 含开头广告
                      - 待删除
```

### 4.2 判定逻辑

| 条件 | 动作 | 目标目录 |
|------|------|---------|
| 视频大小 < 阈值 | 判定为广告/无效文件 | 待删除 或 直接删除 |
| 视频大小 ≥ 阈值 + 命中爬虫番号 | 按番号重命名 | 待整理 |
| 视频大小 ≥ 阈值 + 未命中番号 | 保留原名搬运 | 待整理 |
| 命中开头广告特征（AI/规则） | 标记为开头广告 | 含开头广告 |

### 4.3 AI 广告检测

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ FFmpeg 提取 │───▶│ ONNX 推理   │───▶│ dHash 比对  │───▶│ 风险判定    │
│ 帧采样      │    │ 特征提取    │    │ 样本匹配    │    │ 分流决策    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**三种模型策略**：
| 模型 | 速度 | 敏感度 | 适用场景 |
|------|------|--------|---------|
| MobileNetV3 Lite | 中等 | 中等 | 默认推荐 |
| SqueezeNet Fast | 快 | 低 | 性能弱的设备 |
| YOLOv8n Balanced | 慢 | 高 | 需要严格过滤 |

## 5. Cloudflare 绕过机制

### 5.1 原理
JAV 网站通常有 Cloudflare 年龄验证弹窗，普通 HTTP 请求无法通过。项目使用 Puppeteer 启动用户本地 Chromium 浏览器，模拟真人操作：

1. 启动 Chromium（非内嵌，调用用户系统已安装的 Chrome/Chromium）
2. 访问目标页面，等待 Cloudflare 验证通过
3. 提取通过后的 Cookie
4. 将 Cookie 附加到后续 HTTP 请求中

### 5.2 浏览器查找优先级
```
1. Windows 注册表查找 Chrome 安装路径
2. 常见安装路径扫描：
   - C:\Program Files\Google\Chrome\Application\chrome.exe
   - C:\Program Files (x86)\Google\Chrome\Application\chrome.exe
   - %LOCALAPPDATA%\Google\Chrome\Application\chrome.exe
3. 查找 Chromium 安装路径
4. 均未找到 → 报错提示用户安装 Chrome
```

## 6. 数据流

### 6.1 爬虫数据流
```
用户输入 URL
    │
    ▼
RunnerService.startCrawler()
    │
    ▼
Child Process 启动 → dist/jav.js crawl [options]
    │
    ▼
QueueManager 四阶段流水线
    │
    ├── Phase 1: 索引页 → 提取详情链接
    ├── Phase 2: 详情页 → 提取元数据
    ├── Phase 3: 写入 JSON/CSV
    └── Phase 4: 下载海报图片
    │
    ▼
输出到用户指定目录
    │
    ▼
IPC 推送进度/日志/状态到前端
```

### 6.2 整理数据流
```
用户选择源目录 + 输出目录
    │
    ▼
OrganizerService.startOrganizer()
    │
    ├── scanPhase: 扫描视频文件
    ├── judgePhase: 分类判定
    ├── renamePhase: 重命名 + 移动
    ├── adLearningService: AI 广告检测
    └── reportPhase: 生成报告
    │
    ▼
输出到目标目录（待整理/含开头广告/待删除）
    │
    ▼
IPC 推送进度/日志/状态到前端
```

## 9. 设计原则

1. **代码解耦**：桌面端与抓取核心分离，降低连带风险
2. **保守兜底**：不确定时不丢弃数据，回退到安全策略
3. **可见性优先**：所有操作实时推送日志和进度到前端
4. **本地化存储**：所有配置、日志、学习数据存储在用户本地
5. **沙盒安全**：Electron 启用沙盒模式，通过 preload.js 桥接
6. **可扩展性**：多渠道榜单、多模型策略、多站点解析均预留扩展位
